# PR作成ガイド

## 基本方針

KiroでPRを作成する際は、以下の手順と基準に従ってください。

## PR作成手順

### 1. 事前準備
- 作業内容が明確に定義されていることを確認
- 関連するテストが実装されていることを確認
- コードの品質チェックが完了していることを確認（pint、larastanによる静的解析）

### 2. ブランチ作成
- 基本的に`feature/MDXA-`から始まるタスクIDを使用（NotionページのタスクIDに紐づく）
- 例: `feature/MDXA-1234`, `feature/MDXA-4567-amazon-connect-queue`
- タスクIDが不明な場合は機能名を含む分かりやすい名前を使用

### 3. コミットメッセージ
- 日本語で記載
- 変更内容を簡潔に説明
- 例: `feat: Amazon Connectキュー管理機能を追加`, `fix: ユーザー登録時のバリデーションエラーを修正`

### 4. PR作成時の必須項目

#### タイトル
- NotionページのタイトルをPRタイトルとして使用
- Notionページが見つからない場合は、変更内容から適切なタイトルを作成
- プレフィックスを使用: `feat:`, `fix:`, `refactor:`, `test:`, `docs:`
- 例: `feat: Amazon Connectキュー管理機能の実装`, `fix: ユーザー登録時のバリデーションエラー修正`

#### 説明文
`.github/pull_request_template.md`のフォーマットに従って記載:

```markdown
## Notion のリンク
- 関連するNotionページのURL（あれば記載）

## やったこと
- このプルリクで何をしたのかを具体的に記載
- 追加した機能や修正した内容をリスト形式で記載
- 影響範囲を明記

## やらないこと
- このプルリクでやらないことは何か？（あれば。無いなら「無し」で OK）
- やらない場合は、いつやるのかを明記する

## 動作確認
- どのような動作確認を行ったのか？結果はどうか？
- 実装したテストの内容
- 手動テストで確認した項目と結果

## その他
- レビュワーへの参考情報（実装上の懸念点や注意点などあれば記載）
- 関連するissue番号があれば記載
```

## 品質基準

### テスト
- 新機能には対応するテストを必須で実装
- 既存テストが破綻していないことを確認
- カバレッジの維持・向上


## 自動化ツール活用

- GitHub Actionsによる自動テスト実行を活用
- Pint（コードフォーマッター）による自動フォーマット
- Larastan（静的解析ツール）による品質チェック
- 静的解析ツールの結果を確認してからPR作成

このガイドに従って、品質の高いPRを作成してください。
